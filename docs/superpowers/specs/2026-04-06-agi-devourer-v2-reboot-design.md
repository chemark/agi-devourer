# 《舌尖上的 AGI》V2 重启版设计文档

日期：2026-04-06
状态：已通过脑暴确认，可进入实现规划
范围：彻底放弃旧 V2 实现后，重新定义 V2 Alpha 的产品与技术方向

## 1. 设计摘要

新版 V2 不再以“先堆玩法”作为第一目标，而是改成一个“产品链路优先”的移动端 Web 产品。

第一版要优先打通的不是最复杂的游戏机制，而是这条完整闭环：

1. 用户进入竖屏产品首页
2. 用户以游客身份直接开始挑战
3. 用户被引导进入横屏游戏局内
4. 用户完成一局并看到有情绪价值的结算
5. 若用户未登录，则在结算页被明确引导登录
6. 登录成功后，系统自动补交该局成绩并反馈排名

这一版既不是最终商业成品，也不是纯演示稿，而是一个“产品骨架正确、主链路完整、未来易于接真服务”的 V2 Alpha。

## 2. 产品方向

### 2.1 产品定位

第一版 V2 的定位是：

一个带有街机游戏内核的轻产品化 H5。

优先级排序如下：

1. 产品链路是否清晰
2. 游客、登录、上榜这条转化链是否顺畅
3. 竖屏首页到横屏游戏的切换是否自然
4. 局内玩法是否有基础爽感和复玩欲

这一版不优先追求：

1. 最丰富的玩法系统
2. 最完整的增长传播能力
3. 真正接通后端与微信授权

### 2.2 用户感受目标

用户在第一版里应该自然感受到：

- “我一眼就知道这是干什么的”
- “我不用注册也能马上玩”
- “这局成绩如果不错，我愿意登录去冲榜”

## 3. 平台与横竖屏策略

### 3.1 平台优先级

第一版以移动端 Web 为绝对优先，页面结构优先考虑手机访问体验。

### 3.2 横竖屏分工

新版 V2 采用“产品页竖屏，游戏页横屏”的组合：

- 首页为竖屏产品首页
- 真正的局内玩法为横屏
- 用户不会从首页直接进入游戏局内
- 用户会先进入横屏准备页，再正式开始

### 3.3 旋转引导策略

已确认的策略是：

- 用户从竖屏首页点击“立即挑战”
- 进入横屏准备页
- 准备页明确提示用户旋转设备
- 只有在设备横屏且用户再次确认开始后，才真正开局

这样做的原因是：

- 首页竖屏更适合承接榜单、登录、活动位和产品信息
- 局内横屏更适合双手握持与触控布局
- 中间加入准备页后，横竖屏切换不会显得突兀

## 4. 核心用户流程

第一版 V2 的核心流程如下：

1. 进入竖屏首页
2. 点击“立即挑战”
3. 进入横屏准备页
4. 进入横屏游戏局内
5. 完成结算
6. 若为游客，则触发登录引导
7. 完成登录
8. 自动补交本局成绩
9. 返回上榜成功反馈与当前名次
10. 用户选择再来一局或返回首页

### 4.1 首页目标

首页只承担四类职责：

- 建立产品认知
- 给出明确挑战入口
- 让用户看到竞争感与榜单动机
- 给出登录入口，但不强制先登录

首页不承担以下职责：

- 强行做复杂教程
- 承载过重剧情
- 直接充当游戏局内

### 4.2 结算到登录的转化逻辑

结算页是整条产品链路中的关键转化页。

当用户尚未登录时：

- 系统保留本局成绩为“待提交状态”
- 明确告诉用户“登录后可上榜”
- 登录动作应当被设计成“保存这次高分并冲榜”的自然下一步

当用户登录成功后：

- 应回到当前结果上下文
- 自动发起这条待提交成绩的补交流程
- 明确反馈“上榜成功”与当前排名

## 5. 页面结构与职责边界

第一版只保留 5 个核心页面或层级，不继续扩张。

### 5.1 竖屏首页

职责：

- 展示产品标题与一句钩子文案
- 展示榜单预览
- 展示微信登录入口位
- 提供“立即挑战”主按钮
- 可保留一个轻量活动位或战报位

### 5.2 横屏准备页

职责：

- 明确提示用户横屏
- 简短说明操作方式
- 提供正式开局确认按钮

横屏准备页是竖屏首页与横屏游戏之间的缓冲层，必须保留。

### 5.3 横屏游戏局内

职责：

- 跑完一局 60 秒挑战
- 保持玩法层的专注与手感
- 尽量避免登录、榜单、活动信息打断局内体验

### 5.4 结算页

职责：

- 展示本局分数与结果摘要
- 展示一条简短结果文案
- 说明是否具备冲榜价值
- 在用户未登录时触发登录引导

### 5.5 登录弹层或登录页

职责：

- 解释为什么需要登录
- 完成登录态切换
- 回到当前成绩提交结果

## 6. 第一版玩法设计

### 6.1 核心局内规则

第一版玩法要尽量轻，但要足够支撑产品转化。

核心规则：

- 单局时长 60 秒
- 玩家扮演赛博变色龙，吞噬空中的 AI 模型目标
- 操作逻辑保持简单直观
- 横屏布局要适配双手握持和触控操作

### 6.2 目标种类

第一版只做 3 到 5 种 AI 模型敌人。

它们之间的差异主要来自：

- 分值不同
- 速度不同
- 命中难度略有差异

第一版不引入：

- 深度行为树
- 复杂技能系统
- 过重的特殊机制

### 6.3 难度节奏

整局节奏分成三个阶段：

- 前段：快速让玩家理解玩法
- 中段：明显增加压力和速度
- 后段：制造“差一点就更高分”的张力

这一节奏必须服务结算情绪：

- 输了也要让人觉得想再来一局
- 打得好时要让人觉得“值得登录去上榜”

### 6.4 结果文案

结算页需要有一条“像战报一样”的简短结果文案。

第一版处理方式：

- 产品层保留这项能力的位置
- 实现层先用规则模板或 mock 数据生成
- 不在第一版接真实 AI 文案生成

## 7. 登录与榜单策略

### 7.1 登录规则

已经确认的产品规则是：

- 游客可以直接开玩
- 但若要上榜，必须先登录

### 7.2 登录方式策略

长期目标是微信登录。

但第一版只做到：

- UI 结构、状态模型、服务接口都按未来微信登录来设计
- 实际登录流程先用 mock 方式跑通
- 明确不接真实微信 OAuth 或网页授权

### 7.3 榜单策略

第一版榜单的原则是：

- 体验上像真产品，而不是像静态演示图
- 接口和状态流转按真实读榜、提交成绩来设计
- 但底层先用 mock 数据与本地逻辑完成

这样后续接真后端时，不需要推翻现有页面信息架构。

## 8. 第一版范围控制

### 8.1 第一版必须做

- 竖屏首页
- 横屏准备页
- 横屏游戏局内
- 结算页
- mock 登录流程
- 榜单预览与上榜反馈
- mock 的认证、榜单、结果文案服务
- 游客到登录再到上榜的完整主链路

### 8.2 第一版明确不做

- 真微信登录
- 真后端用户系统
- 真远端排行榜持久化
- 分享海报、二维码传播、朋友圈战报
- 复杂剧情系统
- 局外成长系统
- 过多玩法扩张

## 9. 技术架构建议

### 9.1 推荐架构

第一版推荐采用：

- React + Vite + TypeScript 作为产品壳
- 一个轻量的 Canvas + TypeScript 游戏模块作为局内实现
- Mock Service 层承接认证、榜单和结果文案能力

### 9.2 为什么这一版不先上 Phaser

不是因为 Phaser 不适合，而是因为这一轮重启的重点不是引擎能力，而是结构稳定性。

延后 Phaser 的原因：

- 上一版失败的根本问题不是“引擎不够强”，而是产品链路、玩法、账号、榜单、分享同时变重
- 第一版要优先验证产品闭环，而不是优先构建重型玩法系统
- 先用轻量 Canvas 模块更容易把局内玩法控制在“够玩、够稳”的范围内

如果后续版本确实需要更复杂的碰撞、动画和敌人行为，再升级为 Phaser 会更稳。

### 9.3 模块边界

建议拆成以下边界：

- `app shell`
  管页面结构、横竖屏状态、全局产品流程
- `game session`
  只管一局游戏
- `result flow`
  管结算展示、待提交成绩、登录后补交
- `mock product services`
  管认证、榜单、结果文案接口

其中一个重要原则是：

游戏模块不能直接承担登录和榜单逻辑。

## 10. 核心数据流

第一版必须稳定支持这条数据流：

1. 游客进入首页
2. 游客开始挑战
3. 用户横屏后进入一局游戏
4. 局内产出分数与结果摘要
5. 若用户未登录，则把成绩保存为待提交状态
6. 用户完成 mock 登录
7. 全局登录态更新
8. 系统自动补交待提交成绩
9. 页面反馈当前名次与上榜成功状态

这是第一版最重要的结构主干，后续实现方式可以调整，但这条链路不能丢。

## 11. 错误处理与降级策略

第一版需要明确支持以下降级：

- 登录失败时，不清空当前结果，仍保留重新尝试入口
- 榜单提交失败时，不丢失当前分数与结果页上下文
- 结果文案生成失败时，回退到固定模板文案
- 用户未横屏时，不允许正式开局，但始终给清晰提示与返回路径

## 12. 验收标准

第一版如果达到以下条件，就算成功：

1. 用户进入首页后，能迅速理解这是一个什么产品
2. 游客可以顺畅完成一局
3. 竖屏到横屏的切换看起来是设计过的，而不是坏掉了
4. 结算页能自然推动用户去登录
5. mock 登录后，系统能自动补交成绩并反馈排名
6. 整体感受像一个产品 Alpha，而不是一堆分散页面和游戏片段

## 13. 交付顺序建议

建议后续实现顺序按下面走：

1. 先搭产品壳与横竖屏流转
2. 再做最小可玩的一局游戏
3. 再做结算页与待提交成绩逻辑
4. 再做 mock 登录与 mock 榜单补交
5. 最后做视觉统一与产品感打磨

这样可以最大程度降低“又做重了、但还是不满意”的风险。
